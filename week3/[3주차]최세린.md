## 로또 과제를 하면서 알게된 점


### MVC 패턴
Model, View, Controller로 나누어 설계했다.  
view가 controller을 알고 있는 문제가 발생되었는데, 각각의 부분이 서로 알지 못하게 하는데 더 초점을 두어야겠다.
이 문제를 해결하기 위해 DTO를 사용할 수 있다. DTO는 프로세스 간에 데이터를 전달하는 객체로 데이터를 전달하기 위해 존재한다.
DTO를 사용하면 내부 구조가 바뀌어도 바꿔야하는 외부구조가 적으며 유지 보수에 좋다.

### Controller의 과도한 의존
Controller의 과도한 책임이 발생하였는데, 이를 막기위해 AppConfig을 사용해야한다.
AppConfig은 애플리케이션의 전체 동작 방식을 구성하기 위해, 
구현 객체를 생성하고, 연결하는 책임을 가지는 별도의 설정 클래스이다.
애플리케이션의 전체 구성을 담당해 결합도를 낮출 수 있다.

### Enum
Enum을 활용하여 이번에 코드를 작성하였는데, Enum은 어떨 때 사용하여야는지 알게되었다.
Enum은 열거형으로 관련된 상수들의 집합이며 필드, 생성자, 메서드 정의가 가능하여 행위를 넣을 수 있다.
상수 클래스는 단순 상수들의 나열으로 단순한 값만 저장하며 필드, 생성자, 메서드 정의가 불가능하다. 
따라서 단순한 나열은 상수 클래스 상수들의 행위가 필요하고 관련된 상수들을 모아야한다면 Enum을 사용하면 된다.

### 학습을 통해 느낀점
MVC를 패턴을 학습한지 얼마 안되어서 미숙했고 각 부분의 역할에 대해 다시 생각해보아야겠다.
기능 구현만 신경 쓰는 것이 아닌 그 내부 구조를 어떻게 나누어야하는지에 대해 더 자세히 학습하고 세션에서 배운 내용을 바탕을 코드를 수정하고 추가 내용을 학습해보아야겠다.
